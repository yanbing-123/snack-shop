# 火车票抢票系统 - 需求文档

**版本:** 1.0  
**创建日期:** 2026-03-19  
**状态:** 需求规划阶段

---

## 1. 项目概述

### 1.1 项目背景

随着春运、节假日等高峰期火车票供需矛盾加剧，用户难以通过手动方式及时购买到所需车票。本系统旨在提供一个合法合规的火车票监控与抢票解决方案，帮助用户提高购票成功率。

### 1.2 项目目标

- 提供实时余票监控功能
- 在有余票时自动完成购票流程
- 确保系统稳定运行，避免账号被封禁
- 保护用户隐私和账号安全

### 1.3 合规声明

**重要:** 本系统设计严格遵守以下原则：
- 仅作为用户手动购票的辅助工具，不替代人工操作
- 遵守 12306 官方网站的使用条款
- 不设计恶意刷票、批量占票等违规功能
- 请求频率控制在合理范围内，模拟正常用户行为

## 2. 功能需求

### 2.1 用户登录/注册模块

#### 2.1.1 功能描述
用户通过 12306 账号登录系统，系统不存储用户密码，仅保存加密的登录凭证。

#### 2.1.2 功能点
| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F001 | 12306 账号登录 | 用户使用 12306 官方账号登录 | P0 |
| F002 | 账号绑定 | 绑定用户的 12306 账号（用户名加密存储） | P0 |
| F003 | 登录状态保持 | 保存登录凭证，避免频繁登录 | P0 |
| F004 | 多账号管理 | 支持绑定多个 12306 账号（如家庭账号） | P1 |
| F005 | 乘车人管理 | 添加、编辑、删除常用乘车人信息 | P0 |

#### 2.1.3 业务规则
- 密码不在本地存储，登录时直接提交至 12306 服务器
- 登录凭证（cookie/token）加密存储
- 支持扫码登录和账号密码登录两种方式
- 登录失败 3 次后锁定 15 分钟

### 2.2 车次查询模块

#### 2.2.1 功能描述
用户可查询指定日期、出发地、目的地的车次信息。

#### 2.2.2 功能点
| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F010 | 基础查询 | 输入出发地、目的地、日期查询车次 | P0 |
| F011 | 车次筛选 | 按车次类型（G/D/C/Z/T/K 等）筛选 | P0 |
| F012 | 时间筛选 | 按出发/到达时间范围筛选 | P0 |
| F013 | 席别筛选 | 按座位类型（一等座、二等座、硬卧等）筛选 | P0 |
| F014 | 余票显示 | 显示各席别余票数量 | P0 |
| F015 | 历史查询记录 | 保存最近 10 次查询记录 | P1 |

#### 2.2.3 业务规则
- 查询频率限制：每 3 秒最多 1 次查询
- 支持模糊匹配车站名称
- 查询结果按出发时间排序
- 支持中转方案推荐（二期功能）

### 2.3 余票监控模块

#### 2.3.1 功能描述
设置监控任务，系统自动定时查询余票，有余票时触发通知。

#### 2.3.2 功能点
| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F020 | 监控任务创建 | 设置监控的车次、日期、席别 | P0 |
| F021 | 监控频率设置 | 设置查询间隔（30s/60s/120s/300s） | P0 |
| F022 | 监控时段设置 | 设置监控的起止时间 | P0 |
| F023 | 多任务管理 | 同时监控多个车次/日期 | P0 |
| F024 | 监控状态显示 | 显示当前监控任务状态 | P0 |
| F025 | 智能频率调整 | 根据余票情况自动调整查询频率 | P1 |

#### 2.3.3 业务规则
- 单个账号同时监控任务不超过 5 个
- 查询间隔不低于 30 秒（防封号）
- 监控任务 24 小时后自动过期
- 发现余票后立即暂停该任务监控

### 2.4 自动抢票模块

#### 2.4.1 功能描述
发现余票后自动提交订单，完成购票流程。

#### 2.4.2 功能点
| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F030 | 自动提交订单 | 发现余票后自动填写订单并提交 | P0 |
| F031 | 乘车人选择 | 自动选择预设的乘车人 | P0 |
| F032 | 席别优先级 | 设置多个席别的优先级顺序 | P0 |
| F033 | 支付提醒 | 订单提交成功后提醒用户支付 | P0 |
| F034 | 订单确认 | 提交前可设置是否需要人工确认 | P0 |
| F035 | 失败重试 | 提交失败后自动重试（最多 3 次） | P1 |

#### 2.4.3 业务规则
- 提交订单前必须通过 12306 验证码（支持打码平台）
- 订单提交后 30 分钟内未完成支付将自动取消
- 同一车次同日期最多尝试 3 次
- 抢票失败后自动恢复监控状态

### 2.5 订单管理模块

#### 2.5.1 功能描述
查看和管理已提交的订单。

#### 2.5.2 功能点
| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F040 | 订单列表 | 查看所有订单（待支付/已完成/已取消） | P0 |
| F041 | 订单详情 | 查看订单详细信息 | P0 |
| F042 | 订单同步 | 从 12306 同步最新订单状态 | P0 |
| F043 | 退票/改签 | 支持在线退票和改签（调用 12306 接口） | P1 |
| F044 | 订单导出 | 导出订单记录为 Excel/PDF | P2 |

#### 2.5.3 业务规则
- 订单状态每 5 分钟同步一次
- 待支付订单高亮显示
- 支持按日期、状态筛选订单

### 2.6 通知提醒模块

#### 2.6.1 功能描述
通过多种方式向用户发送通知。

#### 2.6.2 功能点
| 编号 | 功能 | 描述 | 优先级 |
|------|------|------|--------|
| F050 | 余票通知 | 发现余票时立即通知 | P0 |
| F051 | 抢票成功通知 | 订单提交成功时通知 | P0 |
| F052 | 支付提醒 | 订单待支付时提醒 | P0 |
| F053 | 系统通知 | 系统异常、登录过期等通知 | P0 |
| F054 | 多渠道通知 | 支持站内信、邮件、短信、微信 | P1 |
| F055 | 通知设置 | 用户自定义通知方式和时段 | P1 |

#### 2.6.3 业务规则
- 余票通知延迟不超过 5 秒
- 同一事件不重复通知（5 分钟内）
- 夜间模式（23:00-07:00）可设置免打扰

## 4. 接口设计

### 4.1 API 接口规范

#### 4.1.1 接口约定
- 基础路径：`/api/v1`
- 数据格式：JSON
- 认证方式：JWT Token（Header: `Authorization: Bearer <token>`）
- 响应格式：
```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1710835200000
}
```

#### 4.1.2 错误码定义
| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未授权/登录过期 |
| 403 | 禁止访问 |
| 404 | 资源不存在 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |
| 502 | 12306 接口异常 |
| 503 | 服务暂时不可用 |

### 4.3 数据模型设计

#### 4.3.1 用户表 (users)
```sql
CREATE TABLE users (
    id VARCHAR(32) PRIMARY KEY,
    username VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(100),
    phone VARCHAR(20),
    status SMALLINT DEFAULT 1,  -- 1:正常，0:禁用
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
```

## 5. 安全考虑

### 5.1 防封号策略

#### 5.1.1 请求频率控制
| 操作 | 最小间隔 | 说明 |
|------|----------|------|
| 登录 | 60 秒 | 登录失败后延长至 300 秒 |
| 车次查询 | 3 秒 | 随机±20% 抖动 |
| 余票查询 | 30 秒 | 监控任务最低间隔 |
| 提交订单 | 5 秒 | 失败后指数退避 |
| 验证码请求 | 10 秒 | 避免频繁请求 |

#### 5.1.2 行为模拟
- **请求头随机化:** 轮换 User-Agent、Accept-Language 等
- **设备指纹:** 维护固定的设备特征（屏幕分辨率、时区、字体等）
- **操作轨迹:** 模拟鼠标移动和点击延迟
- **时间规律:** 避免固定时间间隔，加入随机因素

#### 5.1.3 异常处理
- 检测到验证码时自动切换打码方式
- 连续失败 3 次后暂停任务 15 分钟
- 账号被限制后自动停止所有任务并通知用户
- 定期更换 IP 地址（使用代理池）

#### 5.1.4 12306 风控规避
```
策略：
1. 单账号同时在线任务 ≤ 5 个
2. 单账号日查询次数 ≤ 500 次
3. 单账号日下单次数 ≤ 20 次
4. 避免凌晨 2:00-5:00 高频操作
5. 新账号前 3 天降低使用频率
```

### 5.2 请求频率控制

#### 5.2.1 限流实现
```python
# 使用 Redis 实现滑动窗口限流
class RateLimiter:
    def __init__(self, redis_client):
        self.redis = redis_client
    
    def is_allowed(self, key: str, max_requests: int, window_seconds: int) -> bool:
        """检查请求是否允许"""
        now = time.time()
        window_start = now - window_seconds
        
        # 移除窗口外的请求
        self.redis.zremrangebyscore(key, 0, window_start)
        
        # 统计窗口内请求数
        request_count = self.redis.zcard(key)
        
        if request_count < max_requests:
            self.redis.zadd(key, {str(now): now})
            self.redis.expire(key, window_seconds)
            return True
        return False
```

#### 5.2.2 限流配置
| 接口 | 限流规则 | 说明 |
|------|----------|------|
| 登录接口 | 5 次/分钟/IP | 防止暴力破解 |
| 查询接口 | 20 次/分钟/用户 | 正常查询频率 |
| 监控任务 | 5 个/用户 | 同时运行任务上限 |
| 下单接口 | 3 次/分钟/用户 | 防止恶意下单 |

### 5.3 用户数据保护

#### 5.3.1 数据加密
| 数据类型 | 加密方式 | 说明 |
|----------|----------|------|
| 密码 | bcrypt | 加盐哈希，不可逆 |
| 12306 账号 | AES-256-GCM | 对称加密，密钥独立存储 |
| 身份证号 | AES-256-GCM | 对称加密 |
| 手机号 | AES-256-GCM | 对称加密 |
| 登录凭证 | AES-256-GCM | 定期轮换密钥 |

#### 5.3.2 密钥管理
```
密钥存储方案:
1. 主密钥存储在环境变量或密钥管理服务（如 AWS KMS）
2. 不将密钥写入代码或配置文件
3. 密钥定期轮换（90 天）
4. 不同环境使用不同密钥
```

#### 5.3.3 数据传输安全
- 全站 HTTPS（TLS 1.3）
- API 请求签名（HMAC-SHA256）
- 敏感接口增加二次验证
- 禁止跨域请求（CORS 白名单）

#### 5.3.4 隐私保护
- 用户数据最小化收集
- 日志脱敏处理（隐藏身份证号、手机号中间位）
- 支持用户数据导出和删除
- 不向第三方共享用户数据

### 5.4 系统安全

#### 5.4.1 认证授权
- JWT Token 认证，有效期 2 小时
- Refresh Token 机制，有效期 7 天
- 基于角色的访问控制（RBAC）
- 敏感操作需要重新验证

#### 5.4.2 输入验证
- 所有输入参数严格校验
- SQL 参数化查询（防注入）
- XSS 过滤（HTML 实体编码）
- 文件上传类型和大小限制

#### 5.4.3 安全审计
- 记录所有敏感操作日志
- 异常行为检测和告警
- 定期安全扫描和渗透测试
- 漏洞响应和修复流程
